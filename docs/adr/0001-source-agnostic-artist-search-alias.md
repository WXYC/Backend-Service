# 0001 — Source-agnostic Artist Search Alias storage

The catalog-side trigram search in Backend-Service does not match an artist filed as "OHSEES" when a DJ types "Thee Oh Sees", because the only data that knows those two strings refer to the same artist lives upstream (Discogs `<namevariations>` in the discogs-cache PG read by LML, and the WXYC `library.alternate_artist_name` column populated by tubafrenzy library staff). We are introducing a local cache table — `artist_search_alias` — populated by a daily ETL pulling from a new LML bulk endpoint plus the local `library.alternate_artist_name` column, and consulted by the catalog search trigram path via a LATERAL JOIN. The keystone decision is that this cache is **source-agnostic**: a single polymorphic table with a `source TEXT` column tagging origin (`discogs_name_variation`, `discogs_alias`, `discogs_member`, `wxyc_library_alt`, future `musicbrainz_alias`, etc.), not three Discogs-shaped tables mirroring discogs-cache's schema.

The LML endpoint that feeds this cache (`POST /api/v1/artists/search-aliases/bulk`) is correspondingly source-agnostic. BS sends WXYC canonical artist names; LML resolves through `entity.identity` to all known external IDs (Discogs, MBID, Wikidata, …), joins to the per-source alias tables in their respective caches, and returns a uniform `{name, variants: [{source, variant, related_external_id?, method, confidence}]}` shape. Adding MusicBrainz aliases later is an LML-only change — no BS schema migration, no BS code change. BS just starts seeing rows tagged `source = 'musicbrainz_alias'`.

## Considered options

- **Three Discogs-shaped tables** (`artist_name_variation`, `artist_alias`, `artist_member`) mirroring the discogs-cache schema. Rejected: locks the BS schema to Discogs's specific kind-of-relationship vocabulary, makes adding a new source (MusicBrainz aliases — which have their own type taxonomy of search-hint / sort-name / legal-name) require either new sibling tables or kind-coercion at the ETL boundary. The kind-discrimination buys cleaner per-table CHECK constraints (e.g. `active` only on members) but those constraints are themselves Discogs-flavored — MusicBrainz aliases don't have `active`, they have `type` and `locale`. The strictness is not earning much.

- **Polymorphic on `kind`, not `source`.** One table with a `kind ∈ {variation, alias, member}` column tagging the relationship type rather than the origin. Rejected for the same reason: the kind taxonomy is Discogs-shaped. MusicBrainz aliases don't fit `{variation, alias, member}` cleanly — a MusicBrainz "search hint" is a variant in the BS sense but not a "Discogs name variation".

- **Migrate `library.alternate_artist_name` into the new table outright.** Drop the column from `library`; route writes from `addAlbum` and `library-etl` directly to `artist_search_alias`. Rejected: the column is upstream-curated through tubafrenzy and the legacy MySQL ETL; removing it requires cross-repo coordination tied to the [tubafrenzy turndown](https://github.com/orgs/WXYC/projects/) timeline, which is on life support with no firm decommission date. Shadow-ingest (mirror the column into `artist_search_alias` via the daily ETL) keeps the column where it is and is reversible.

- **Build the cache against LML's existing per-artist endpoint** `GET /api/v1/discogs/artist/{id}`. Rejected: every cold artist is one Discogs-API call under the LML `Semaphore(5) + 50/min` chokepoint. Warming the WXYC catalog this way is the exact cascade-cascade saturation pattern that pinned [LML#370/#372](https://github.com/WXYC/library-metadata-lookup/issues/370) in the 2026-05-24 incident. A cache-only bulk endpoint reading directly from discogs-cache PG (and musicbrainz-cache PG, etc.) avoids the Discogs-API chokepoint entirely.

## Consequences

- **BS owns its key-space.** The cache is keyed on WXYC `artists.id`, mirroring the post-#800 `library_identity` / `library_identity_source` convention (BS as thin writer of LML-composed verdicts; external IDs stored as audit columns, not join columns). Read-path JOIN is one hop: `artist_search_alias.artist_id = library.artist_id`.

- **Future sources are purely additive.** Adding MusicBrainz aliases is one new `source` value emitted by LML, no BS schema migration. Same for Wikidata labels, Bandcamp tags, or any future origin. The cache's storage shape is fixed.

- **Per-source CHECK constraints are loose.** `active` is only meaningful for `discogs_member` rows; `related_artist_id` and `external_object_id` are only meaningful for relationships pointing at other artists (`discogs_alias`, `discogs_member`). The schema does not enforce these conditional invariants; instead the BS ETL extractor sets them appropriately per source. Trade-off accepted because the constraint vocabulary is itself source-flavored — coercing all sources into a single CHECK lattice would either be permissive enough to be near-useless or restrictive enough to block a future source.

- **Per-source search-ranking is a v2 capability.** v1 search ranks all variants by raw `similarity()` without per-source weights. The `source` and `method` columns are stored so v2 can layer weighting (e.g. `wxyc_library_alt > discogs_name_variation > discogs_alias > discogs_member`) once observed precision data exists. No schema migration needed for that.

- **The LML endpoint must compose, not just dispatch.** `POST /api/v1/artists/search-aliases/bulk` cannot be a thin wrapper around the discogs-cache `artist_name_variation` table — it has to resolve each input name through `entity.identity` (with the same case-insensitive + canonicalization fall-through that `bulk-resolve-libraries` uses), then fan out to all source caches (discogs-cache today; musicbrainz-cache when MB aliases ship), then merge results with source-tagging. This is a non-trivial new LML composer, analogous to the `identity/bulk_resolve.py` cross-cache-identity composer.

- **Flowsheet search alias-awareness is deferred.** v1 plumbs alias search through `library.artist_id` only. `flowsheet` has no `artist_id` column today, only `album_id` (~46% populated) and free-text `artist_name`. Alias-aware flowsheet search depends on a separate v2 epic that adds `flowsheet.artist_id` and backfills the ~1.06M unlinked rows using the alias cache itself as one of the resolvers. The v2 work is unblocked but not in v1 scope.
