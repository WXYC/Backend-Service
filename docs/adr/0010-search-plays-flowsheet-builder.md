# 0010 — `POST /flowsheet/search` endpoint with structured body and always-included histogram

The iOS DJ tool adds Search Plays — a new top-level Tab that searches the flowsheet archive (back to Nov 2004), parallel to its existing catalog search. We expose one new endpoint: `POST /flowsheet/search` taking a structured JSON body (filter rows, sort, pagination) and returning paginated `FlowsheetV2TrackEntry[]` results plus an always-included `year_counts` / `month_counts` histogram for the matched set. This serves iOS's row-based builder UI directly — the iOS app composes filters via a structured builder sheet rather than a query string, and we accept the structure as-is rather than maintain a wxyc.info-style text-syntax parser that iOS doesn't need.

Compared to the existing wxyc.info playlist-search backend (which iOS would otherwise have mirrored), this endpoint adds first-class date and date-range filters, returns structured filter metadata that round-trips cleanly through the iOS APIClient, and unifies pagination with the cursor-based pattern we already use elsewhere. The histogram side-channel mirrors wxyc.info's existing `yearCounts` field that ships on every response — including the top-10k-by-relevance cap with a footer note when `totalHits > 10000`.

Canonical source: [`wxyc-dj-tool-ios/docs/cross-repo-adrs.md` ADR 0009](https://github.com/WXYC/wxyc-dj-tool-ios/blob/main/docs/cross-repo-adrs.md#adr-0009--flowsheet-archive-search-is-a-distinct-ios-mode-with-a-reusable-structured-filter-builder) and the repo-local [iOS ADR 0004](https://github.com/WXYC/wxyc-dj-tool-ios/blob/main/docs/adr/0004-search-plays-flowsheet-builder.md).

## Our side of the work

- **Add `POST /flowsheet/search`** to a new or existing flowsheet controller in `apps/backend/src/`. Request body: `{ filters: [{field, op, value, exact, valueTo?}, ...], sort: {field, descending}, page: int, pageSize: int }`. Field set: `artist` / `song` / `album` / `label` / `dj` / `date` / `date_range`. Operators between rows: `AND` / `OR` / `NOT`. Per-row `exact` boolean on text fields (false = substring, true = exact phrase). Response: `{ results: FlowsheetV2TrackEntry[], totalHits, year_counts, month_counts }`. Histogram is always included — no opt-in flag — so a client that forgets the flag doesn't silently lose the headline feature.
- **Top-10k cap on the histogram bucketization** when `totalHits > 10000`. Response includes a `histogram_truncated: true` flag (or equivalent) so the iOS client can render the footer note "Distribution of top 10,000 results by relevance" verbatim with wxyc.info.
- **Cursor-based infinite-scroll pagination** at 100 rows per page, matching the pattern we already use for `/djs/{id}/plays` (ADR 0007). Cache key includes the page so cache hits respect pagination state.
- **60s TTL cache** keyed on a hash of `(filters, sort, page)`. Same staleness posture as ADR 0007 and ADR 0009.
- **Full-text indexing on `flowsheet_entries`** text columns — likely already in place to back wxyc.info's existing search. Confirm coverage on artist / song / release / label / dj fields. If gaps exist, add the missing indexes as part of this work.
- **OpenAPI surface** in [`wxyc-shared/api.yaml`](https://github.com/WXYC/wxyc-shared/blob/main/api.yaml) — `POST /flowsheet/search` path + `FlowsheetSearchRequest` / `FlowsheetSearchResponse` / `FilterRow` / `FilterField` enum / `FilterOperator` enum / sort schema. Documented well enough that iOS's APIClient and dj-site's future migration consumer can both code-generate from `api.yaml` if/when the codegen pipeline expands.

## Consequences for us

- **No new schema.** Endpoint reads existing `flowsheet_entries` with existing full-text indexes. No migration.
- **POST-for-read is intentional.** Search-with-complex-filters is a recognized REST-ish pattern (Elastic, GraphQL gateways, GitHub's `POST /search/issues` for advanced queries). The iOS APIClient already POSTs (sign-in, bin mutations, condition reports per ADR 0005); adding a POST search method is conventional in that codebase. Trade-off accepted: HTTP caching layers that key on the URL alone will miss; we mitigate with the in-application response cache.
- **No grouping or parentheses in the filter grammar.** Rows evaluate linearly left-to-right with their AND/OR/NOT operators. Matches dj-site's `PlaylistAdvancedSearch.tsx` semantics exactly. Avoids the recursive-AST complexity that a parenthesized grammar would require server-side.
- **Future dj-site convergence opportunity.** dj-site's playlist search currently runs against its own backend with its own text-syntax shape. Once iOS ships, we expect dj-site to file a future ADR migrating its `PlaylistAdvancedSearch.tsx` consumer onto `POST /flowsheet/search` — retiring the legacy backend and converging on one endpoint serving both surfaces. Not committed in this ADR — flagged here so we don't accidentally diverge the endpoint shape from what dj-site will eventually need.
- **Histogram payload is small.** Max ~280 integers per response (~22 years + ~252 months since Nov 2004) is well under 5 KB. The 100-row pagination payload dwarfs it. Always-included is the safer default.
- **The endpoint replaces zero existing endpoints.** wxyc.info's current playlist-search backend stays exactly as-is (it serves wxyc.info, not the new iOS Search Plays). If dj-site eventually migrates, we'd retire its backend in that follow-up.
